# Agent 稳定币支付通道

**长期命题**

为 AI agent 提供一条可"**自主发起、自主接收、自主确认**"的稳定币支付通道。定位为"**面向 agent 的 Stripe**"——仅承担支付层职能（确认款项到账并通知交易双方），不承担履约（服务交付由商户负责）。

**Phase 1 交付（收敛）**

**非托管支付中介 + 风控 + 对账 + 制裁筛查**：让程序对程序的稳定币支付付得安全、对得上账、尽量不用去办那些难拿的金融牌照。（"支付中介"只验签、广播交易，不碰资金，详见术语表）

三项完整"自主能力"属长期愿景；一期"自主接收"只做到"**agent 发起 + 收款方确认到账**"，完整 agent↔agent 自主收款留后（见第 3.3 节）。

---

> **文档结构**：第 0–3 节为命题本质、产品定位与能力范围；第 4 节起为联网查证的可行性分析（竞品格局、安全短板、技术门槛、商业可行性与合规判断）。
>
> **方法说明**：可行性部分结论经多来源联网调研 + 对抗式交叉核验（每条关键主张三方独立验证、多数证伪即剔除），仍属 **2026 年年中的时点快照**，数据具时效性。

## 术语表（先读）

全文术语较多，先用一句话解释常见词，读正文时可回查。

**支付与架构**

| 术语 | 一句话解释 |
|---|---|
| **支付中介**（业界常称 facilitator） | 帮 agent 完成支付的"中介服务"：只对用户签名的付款授权做**验签 + 广播上链**，本身不碰资金。本产品定位即一个非托管支付中介 |
| **非托管（non-custodial）** | 平台**不持有**用户私钥/资金，钱直达对方地址、平台不经手 |
| **托管（custodial）** | 服务方**持有或控制**用户资金（如交易所钱包） |
| **稳定币（stablecoin）** | 与法币锚定、币值稳定的加密货币；**USDC / USDT** 是两大主流美元稳定币 |
| **对账（reconciliation）** | 把每一笔链上到账，精确对应到"谁、为哪次调用、付了多少" |
| **微支付（micropayment）** | 极小额支付（几分钱到几美元），信用卡通道因固定手续费做不了 |
| **on/off-ramp（法币出入金）** | 稳定币 ↔ 人民币/美元的兑换提现与充值 |

**协议与技术**

| 术语 | 一句话解释 |
|---|---|
| **x402** | Coinbase 的开放支付协议：复用 HTTP "402 需付费"状态码，让程序调 API 时自动付稳定币 |
| **AP2（Agent Payments Protocol）** | Google 主导的 agent 支付**授权**协议，定义"这笔钱谁授权、能否抵赖" |
| **Mandate** | AP2 里那张**签名的电子授权书**（证明支出经用户授权、在范围内） |
| **EIP-3009 / Permit2** | 以太坊上的**签名授权转账**标准：用户签个名，别人就能按授权移动其资金（且可代付 gas） |
| **ERC-4337 / 智能账户 / 会话密钥** | "账户抽象"技术，让程序化钱包按预设策略自动签名、但受限额约束 |
| **Base / Solana / Tron** | 公链（跑交易的区块链）；Base 是本产品拟先落地的链 |
| **gas** | 链上交易的手续费 |
| **预确认（pre-confirmation）** | 链上先给的"大概率会成功"的快速承诺，**不是**最终不可撤销的确认 |
| **提示注入（prompt injection）** | 把恶意指令藏进 agent 会读到的内容里，诱导它执行——agent 支付的主要攻击手段 |

**合规**

| 术语 | 一句话解释 |
|---|---|
| **货币转移商（money transmitter）** | 帮他人搬钱的中介，通常须持牌；是否属于此类**看"是否控制资金"**，非看技术 |
| **VASP** | 虚拟资产服务提供商——多国的加密业务发牌类别 |
| **MSB** | 美国的"货币服务商"牌照类别 |
| **KYC / AML** | KYC=客户身份识别；AML=反洗钱 |
| **制裁筛查（OFAC）** | 核对交易地址是否在美国等制裁名单上（一期低成本底线） |
| **KYT（Know Your Transaction，"了解你的交易"）** | 追查一笔链上交易或一个地址的**资金从哪来、干净不干净**——它有没有沾过被盗资金、诈骗、赌博、暗网、混币器等，据此给出风险评分。比"制裁筛查"（只查是否命中名单）范围宽得多，也贵得多（付费产品，年费数十万欧元级），属一期之后的增强项 |
| **FinCEN** | 美国财政部下负责反洗钱监管的机构 |
| **MiCA / CASP** | MiCA=欧盟加密统一监管框架；CASP=其下的加密服务商牌照 |
| **GENIUS Act** | 美国的稳定币立法 |

## 0. 问题定义

传统支付（无论刷卡还是现有加密收款）在设计上均以**人**为操作主体：

- 需人工点击"确认支付"；
- 需弹窗、倒计时、防误操作等机制（"金额不足/超额/迟付"等异常处理，本质是为人工操作的容错）；
- 需人工监看到账，再手动放行或发货。

**AI agent 既不需要、也无法使用上述机制。** 当 agent 需要购买 API 额度、算力、数据，或向另一个 agent 付费时，交易链路中**没有人工环节**。因此本命题的核心不是"改进收款引擎"，而是：

> **将支付从"人工确认"模式，改造为"程序依授权自动完成"模式。**

这即是"自主发起、自主接收、自主确认"三项能力的完整含义。

**能力边界（须明确）**：三项能力均限于**资金层**——发起的是付款、接收的是款项、确认的是到账。**我方不承担履约**：款项到账后是否放行 API、是否交付数据、是否判定订单完成，均属商户的业务逻辑。此定位与 Stripe 一致——Stripe 负责收款与到账通知，发货由商户自理。据此，本产品是**支付基础设施**，而非"agent 服务市场"或"结算履约中台"。

## 1. 时机判断

以下三项条件同时成立，此通道方具备可行性：

1. **agent 已产生真实付费行为**。agent 调用外部 API、算力、数据均为付费动作，且具有**高频、小额、程序触发**的特征，人工逐笔确认不可行。
2. **稳定币天然适配程序化支付**：可编程、7×24 运行、无国界、无最低手续费门槛、逐笔链上可验证。信用卡通道无法承载"每次 API 调用自动支付 0.001 美元"这类微支付。
3. **协议层已初步成型**（x402、AP2，详见第 5 节）。2025 年以前 agent 付费缺乏标准，现已有雏形。**该窗口正在开启，但同时正被巨头快速占据。**

## 2. 能力拆解

将三项能力转化为具体建设内容：

| 能力 | 所需建设 | 与"人工支付"的本质差异 |
|---|---|---|
| **自主发起** | ①钱包/账户；②授权边界——限定资金来源、单笔上限、总额上限、用途范围、有效期 | 无人工确认环节；风控依赖**事前授权与事中限额**，而非事后弹窗 |
| **自主接收** | ①可编程收款地址；②机器可读凭证，标明付款方、对应服务、金额 | 收款方也可能是另一 agent；对账不依赖人工填写订单号，而依赖协议自带 ID |
| **自主确认** | 无人值守地验证到账，并以机器可读方式（回调/协议响应）通知双方 | 无人工监看；我方负责确认并送达"付款完成"这一事实，后续放行与交付属商户履约逻辑，不在我方职责内 |

**须注意的优先级校正**：加密收款中"金额不足/超额/迟付/误操作"等异常处理，主要为人工操作容错而设计。**agent 付款金额由程序精确计算，此类异常将大幅减少**，不应列为首要优先级。agent 场景真正高频的异常属另一类：**授权过期、超限被拒、agent 被劫持导致异常支出、以及退款需求**。（退款方面，我方提供执行通道；"服务是否交付、是否应退"由商户判定并发起，我方不作裁决。）

## 3. 产品定位：程序对程序的非托管结算与风控层

> **一句话定位**：**不做"给所有商户收款"，只做"程序对程序（agent ↔ 服务）的高频付费"这一场景的非托管结算 + 风控 + 对账层。**
> "面向 agent 的 Stripe"仅用于帮助理解；须清醒认识到：**收款本身已不是卖点**（x402/Coinbase 已能直接收稳定币，Stripe 也已部分占位），**我们卖的是它们之上、目前还没人做好的那一层——安全、风控、对账、非托管**。定位若停留在"又一个收款网关"，会被一句"这不就是 Stripe/Coinbase 的子集吗"直接打回；必须收窄到上面这句。

**关于"非托管"的双重角色**：非托管在本产品里**既是产品原则、也是合规支点**——作为产品原则，它让资金直达对方、平台不成为信任瓶颈；作为合规支点，它是规避货币转移/支付牌照的关键（见第 9、13 节）。两者一致，不冲突。

> **落地场景（潜在市场，非收敛客群）**：任何"程序高频付费购买数字服务"的场景都是潜在市场（AI API、算力、数据、SaaS、agent 间协作等）。转售 AI API 的中转站曾被视为候选切入点，但它只是早期场景之一、并非命题本身，且涉及合规与客群质量问题（见第 8、9 节），不作为核心市场。

### 3.1 谁付款、谁收款、谁集成、谁担责（角色界定）

**"agent 是核心用户"容易和"谁签约、集成、担责"混淆——二者必须分开。** agent 是**使用者/发起方**，但真正签约、接 SDK、处理回调、承担风控与合规义务的，通常是平台或开发方。四类角色：

| 角色 | 是谁 | 承担什么 |
|---|---|---|
| **付款方 agent（payer agent）** | 自主发起支付的 AI agent | 触发付款、持钱包与授权边界 |
| **收款服务方（service provider）** | 提供 API/算力/数据、收款的一方（可能也是 agent 或平台） | 提供服务、接收款项 |
| **集成方/平台（integrator/platform）** | 接入本产品 SDK 的 API 聚合器、agent 平台、开发团队 | **实际集成、处理回调、落地风控——通常即签约付费的"客户"** |
| **合规责任主体（customer of record）** | 依辖区承担 KYC/KYT/制裁筛查义务的一方 | 承担反洗钱与合规义务（非托管下义务可下移至此方，见第 13 节 C、D 的行业实践） |

> **关键结论**：**"用户"是 agent，"客户"多半是平台/开发方**。这直接决定：控制台给平台用、退款由收款服务方发起、KYT 义务落在合规责任主体、销售对象是平台而非单个 agent。第一批付费客户最可能是 **API 聚合器 / 模型平台 / agent framework**，而非终端 agent 本身——这一判断须在 GTM 中验证。

**"收款服务方"和"集成方"是两回事，可能是同一家、也可能不是同一家**：前者是"谁提供服务、谁收到钱"，后者是"谁接入我们、谁签约付费"。

- **同一家（直连）**：一个 API 商自己接 SDK、也自己收钱——两个身份是同一家。
- **不是同一家（平台/聚合）**：平台接入我们（集成方），平台下游几十个供应商各自收钱（收款服务方）。**此时一期就需支持"单集成方 + 多收款方"的收款归属映射与对账**（即把每笔到账/回调正确归属到对应收款方，钱仍直达各方地址；**不涉及资金路由/归集/代收再分发**，那属托管红线、明确不做）——这是范围与复杂度的关键分叉（见第 3.3 节）。

**责任边界：我方"提供工具"，而非"承担运营责任"（须在合同与产品上明确划清）**

| 事项 | 我方（工具提供方） | 集成方/平台 | 收款服务方 |
|---|---|---|---|
| 制裁名单筛查 | **提供**地址命中信号（低成本，一期做） | **决定**处置、并负法定责任 | — |
| 完整 KYT（风险评分/来源分析） | **可选提供**（后续增强，高成本） | **决定**是否采用及阈值策略 | — |
| 风控（限额/熔断） | **提供**规则引擎与执行 | **配置**规则、对其业务后果负责 | — |
| 对账 | **提供**到账↔调用的匹配数据 | 使用数据入账 | — |
| 回调通知 | **提供**签名回调与主动查询接口 | 消费回调、驱动自身业务 | — |
| 退款 | **提供**原路退回的执行通道 | 把退款请求转给我方 | **判定**是否该退、并发起 |
| 资金保管 | **不碰**（非托管） | — | 资金直达其地址 |
| KYC（如需） | 一般不做 | 依辖区可能须做 | 视角色可能须做 |

> **一句话划界**：**我方只提供"能力/工具"，不做"应不应该"的业务与合规判断**——阈值、处置、退款与否、KYC 由集成方/收款方承担。此划界既是产品边界，也是"非托管、不承担运营责任"这一合规立场的一部分（呼应第 13 节红线：一旦我方替客户做实质资金/合规决策，就可能被认定越出工具提供方身份）。

### 3.2 为什么"x402/Stripe/Coinbase + 自建"不够（我们凭什么单独做）

需求侧已证明"稳定币支付存在、有人用稳定币买 AI 服务"，但这**不等于**"必须单独做一个 agent 支付通道"。真正要回答的是：现成组合卡在哪？**答案：缺的不是"收款能力"，而是"安全 + 风控 + 对账 + 非托管"的组合层**——收款这一步已被巨头解决，痛点在它之后。具体场景：

| 场景 | 现成方案（x402/Stripe/Coinbase + 自建）卡在哪 | 本产品补的那一层 |
|---|---|---|
| agent 被提示注入劫持、疯狂付款 | x402 只管验签广播、不管"该不该付"；限额/熔断要自建 | 消费限额 + 异常熔断 + 审计 |
| "服务已交付但链上结算被回滚"（x402 已实测 5.18%） | 现成通道不做"结算最终性确认再放行" | 确认钱真的到账、再通知商户放行（含防抢占） |
| 平台要把上万笔微支付对应到具体调用/客户 | 现成通道给的是链上流水，不给业务级对账 | 每笔到账 ↔ 调用/订单精确对账 |
| 平台需对付款地址做制裁/风险筛查 | x402/Coinbase 默认不替你做筛查 | 入账关键路径的制裁名单筛查（一期）；完整 KYT 为后续增强 |
| 想用非托管但又要合规确定性 | 自建难保证"非托管红线"不被无意越过 | 把非托管红线固化进产品（见第 13 节） |

> 一句话：**收款本身谁都能做、不构成壁垒；真正难做、也是我们要立足的，是"让 agent 的钱付得安全、对得上账、又不用去办那些难拿的金融牌照"这一层。**

### 3.3 Phase 1 范围：做什么、不做什么（收敛一期）

当前命题横跨支付通道、钱包授权、风控中间件、对账 SaaS、法币出入金——那是**战略地图**，不是**一期定义**。收敛建议如下（倾向：**支付中介 + 风控 + 对账**为第一版最小可卖组合）：

| 能力 | Phase 1 | 理由 |
|---|---|---|
| 非托管支付中介（验签 + 广播 + 到账确认，照抄 x402 设计） | ✅ 做 | 基座，且合规最干净 |
| 支出限额 / 熔断 / 审计（风控） | ✅ 做 | 差异化核心，巨头没做好 |
| 业务级对账（到账 ↔ 调用/订单） | ✅ 做 | 平台刚需，与风控共用底层 |
| **单集成方 + 多收款方**（每收款方独立地址、独立对账/回调归属） | ✅ 做 | 平台/聚合场景（集成方≠收款方）的刚需；仅做"归属与对账"，资金仍直达各收款方地址 |
| **平台内资金路由 / 归集 / 代收再分发** | ❌ 不做 | 一旦归集即触碰托管红线（见第 13 节 A）；分账需求交由链上直达或持牌方 |
| **制裁名单筛查**（付款/收款地址是否命中 OFAC 等制裁名单） | ✅ 做（接第三方，见第 13 节 C） | 合规底线，成本低（Chainalysis 免费预言机起步） |
| **完整 KYT**（风险评分、资金来源分析、行为监测） | 🟡 后续增强（须单独评估成本） | 范围更宽、成本显著更高（付费 KYT/TRM/Elliptic，年费数十万欧元级），不宜与低成本的地址筛查混为一期同一项 |
| 钱包 / agent 授权层 | 🟡 评估（可先集成 Cobo 等现成方案，不自建） | 有现成参照，非一期自研重点 |
| 法币出入金 | ❌ 不做（挂靠持牌方，见第 12 节） | 牌照负担最重，规模验证后再评估 |
| 履约 / agent 服务市场 | ❌ 不做 | 明确不在职责范围 |

> **关于"单集成方 + 多收款方"的拍板**：Phase 1 **支持**该拓扑，但**只做"多收款方的归属与对账"，不做平台内资金路由/归集/代收再分发**——每个收款方的钱始终直达其自有地址。这样既满足平台/聚合客户的刚需（第 3.1 节的分离场景），又守住非托管红线（归集即托管）。

> **一期卖的是**：**非托管支付中介 + 风控 + 对账 + 制裁筛查**——让程序对程序的稳定币支付付得安全、对得上账、尽量不用去办那些难拿的金融牌照。**"自主接收"在一期可先做到"agent 发起支付 + 收款方确认到账"，完整的 agent↔agent 自主收款留待后续验证**（见第 11 节待决策事项）。

## 4. 技术可行性与成熟度

**结论：技术可行，且已有生产级验证。** 构建 agent 支付通道所需的四项基础组件，均已具备成熟实现：

| 组件 | 简释 | 现状 |
|---|---|---|
| 可编程钱包 | 供程序按规则自动付款、并可设定支出上限的钱包 | 已有成熟产品（详见第 7 节 Cobo） |
| HTTP 内嵌收费 | 使程序在发起网络请求时同步完成付款，无需预先注册或绑卡 | x402 已上线并规模化 |
| 链上即时结算 | 款项在链上数百毫秒内到账，无需银行 T+1 周期 | 已实现（约 200 毫秒，见下方说明） |
| 可签名授权凭证 | 机器可读、防篡改、可验真伪的电子授权凭证 | AP2 已定义 |

**成熟度须分两层评估，此为全文最关键的判断：**

- **协议层：已成型可用。** x402 于 2025 年 5 月上线，至 2026 年初已在 Base 链处理 **1.19 亿笔交易、年化约 6 亿美元**，非试验性质。AP2 于 2025 年 9 月发布。
- **安全层：显著不成熟（重点）。** 2026 年多篇学术论文实证了大量真实漏洞（见第 6 节），且**尚无任何现有协议能覆盖从"AI 决策付款"到"链上结算完成"的完整安全链路**。

> **两项数据须审慎采信**：①"约 200 毫秒到账"实为**预确认**（链上给出"高概率成功"的承诺），并非最终不可撤销的确认——此为第 6 节"服务已交付但款项可能被撤回"漏洞的根源；②"单笔成本低至 0.001 美元"出自 x402 自有白皮书，属利益相关方自述，需独立验证。

## 5. 竞品格局：巨头分三层布局

核心发现：**当前格局并非分散的初创竞争，而是巨头已按三层结构分工布局**，且各层相互配合。

```text
通信层  A2A —— 使不同厂商的 agent 可互相通信（Google 发起，已捐赠 Linux 基金会）
授权层  AP2 —— 界定"支出由谁授权、能否抵赖"（Google 发起，交 FIDO 联盟标准化，60+ 机构参与）
结算层  x402 —— 界定"资金如何以稳定币流转"（Coinbase 主导）
```

**关键：AP2 与 x402 为上下游协作关系，非竞争关系。** AP2 负责授权（签发签名的电子授权凭证），x402 负责资金流转（稳定币结算），且 **x402 是 AP2 唯一的稳定币结算方**。

| 竞品 | 业务 | 是否托管资金 | 目标对象 | 进展 |
|---|---|---|---|---|
| **Coinbase x402** | 为网络请求增加"先付费后交付"的收费环节，agent 调用 API 时以稳定币自动付款 | **否**（款项由付款方钱包直接签发，x402 仅验签并触发上链） | AI agent（买方）+ 拟以 API 变现的服务商（卖方），正对应 API 转售生态的供需两端 | 已规模化：1.19 亿笔 |
| **Google AP2** | 定义 agent 支出的电子授权凭证格式，界定授权与抵赖问题 | 否（协议本身不接触资金） | 全行业（通用标准） | 2025-09 发布，60+ 机构参与（万事达、PayPal、美国运通、Coinbase、以太坊基金会） |
| **Stripe** | 同时支持 x402 与自研 MPP，两条路线并行 | **较可能托管**（"Stripe x402 为非托管"一说经核验以 0:3 证伪） | 企业客户（当前限美国） | 2026-02 上线预览版 |

## 6. 竞品的共性短板（即差异化切入点）

**所有竞品存在同一共性短板：协议推进快，安全建设滞后。** 此结论由 4 篇独立的 2026 年学术论文实证，为全文证据最充分的一组（每条均以 3:0 通过核验）。

- **AP2 授权凭证可被重放与篡改**：已使用的授权凭证可被再次提交，导致**重复扣款**（凭证未设计为一次性失效）；授权凭证与交易未绑定，攻击者可**篡改收款方或金额**。红队研究以"间接提示注入"（将恶意指令植入 agent 会读取的内容）攻击 AP2，**成功率达 100%**。
- **x402 存在"先交付服务、后结算"的信任缺口**：HTTP 层为"付款后立即交付服务"，而链上结算为**异步且仅概率性最终确认**，可能出现**服务已交付、款项却被撤回**、商户蒙受损失的情形。实测该"服务已交付但收不到款"概率为 **5.18%**，极端条件下达 **100%**。
- **x402 付款凭证可被抢先占用**：授权未绑定使用方，攻击者观察到付款签名后可**抢先结算并占用该笔款项**（测试网已实证）。
- **持有自有钱包的自主 agent 风险高**：可自动签发付款、且无逐笔人工审批的 agent，一旦被提示注入劫持，可能被诱导**直接签发真实资金交易**。
- **无任一协议覆盖完整链路**：2026 年 5 月一篇综述论文实测主流协议（ERC-8004、AP2、x402、MPP、OKX 方案），结论为**无任何协议能从"AI 决策"到"结算完成"实现端到端安全，各方案互补但单独均不充分**。

## 7. 差异化切入点与技术门槛

**核心判断：不应再造协议（标准已被巨头联盟锁定），而应建设协议之上的"垂直整合层"。**

本产品差异化由三项能力构成，**各自对应上述一处安全空白**：

1. **安全防护层**：防重放、防抢占、"待链上真正结算确认后再放行"。
2. **支出限额与风控层**：单笔/单日/总额上限、异常自动熔断，以应对 agent 失控风险。
3. **对账层**：将每笔链上到账精确匹配至"付款方、对应调用、金额"，为高频微支付的刚性需求。

叠加两项巨头未覆盖的定位：**非托管**（Stripe 亦未占据）+ **专注"程序对程序（agent ↔ 服务）的高频小额支付"这一通用场景**（API 转售/聚合只是其中一个早期验证场景，非核心定位——与第 3 节口径一致）。

**技术门槛清单（多数具备现成参照，非从零构建）：**

| 门槛 | 简释 | 难度 | 现成参照 |
|---|---|---|---|
| agent 身份与授权 | 证明 agent 有权支出且未被冒充 | 中 | 可直接采用 AP2 授权凭证机制 |
| 可编程钱包与限额 | 供程序按规则自动付款并设定硬性上限 | 中 | **Cobo Agentic Wallet**：以门限签名（多方分持一把私钥）实现默认非托管，设 agent 自动组与人工审批组双签名组，仅人工可完整恢复私钥，内置预算/白名单/审批/审计 |
| 链上监听与对账 | 监听到账并匹配至订单 | **高（最难）** | 需自建，业内多用"订阅式监听 + 每单专用地址" |
| 微支付/流式付款 | 逐次调用逐次付款，或按用量持续扣费 | 中 | 主流不逐笔上链（成本过高），而采用**预授权额度 + 持续小额扣费 + 累积批量结算** |
| 支出限额与风控 | 防止 agent 失控或被劫持后异常支出 | **高（差异化核心）** | Cobo 策略引擎可参照；但"防提示注入"全行业尚未解决，此处存在差异化空间 |
| x402/AP2 兼容 | 接入现有生态、即接即用 | 中 | 直接实现协议即可，文档完备 |

> **关于"非托管"表述的界定**：厂商所称"非托管"往往为营销框架——x402 托管版结算最终进入 Coinbase 商业账户；Cobo 亦持有部分私钥份额，并对每笔自动交易联签。即"资金不完全经我方控制"成立，但"完全无中间方"不成立。产品设计时须明确界定本产品所采用的"非托管"具体形态。

## 8. 商业可行性（联网查证）

**结论：agent 支付需求真实但尚小，盈利模型薄弱。**

**① 核心需求真实：agent 稳定币支付已产生真实（虽小）交易量。** 2025.5–2026.4 期间，AI agent 结算金额超 **7300 万美元、约 1.76 亿笔链上交易**，约 98% 使用 USDC，**均笔约 0.31–0.48 美元**，其中 **76% 低于信用卡常见的 0.30 美元固定手续费门槛**——即属**信用卡通道无法承载、仅稳定币可承接的微支付**，构成 agent 支付通道的天然场景，直接印证本产品命题的需求侧成立。**须注意**：上述数据来自单一做市商报告（Keyrock，对"加密通道胜出"叙事存在利益关联），无独立佐证，且仅统计"已选择链上结算"的交易，存在选择偏差；且当前总规模仍小（全年数千万美元级），属早期。

**② 正规市场已有硬证据，且已在用稳定币收款——这是需求侧最有力的一条。** 以 **OpenRouter**（VC 支持的正规 AI API 聚合器，非灰市）为代表：约 800 万注册用户、每月约 100 万亿 token、13 亿美元估值、约 5000 万美元年化营收，盈利模式为"在底层模型成本上加价约 5% 转售"（Requesty 同为 5%）。**关键**：OpenRouter 收款虽以 Stripe 卡/法币为主，但**已上线加密支付 API，支持 USDC 链上支付（Base/Ethereum/Polygon，由 Coinbase Developer 驱动）**——这直接证明"正规 AI API 市场对稳定币收款存在真实需求且已落地"。**但须注意的竞争含义**：这类头部聚合器已能直接对接 Coinbase 等现成通道，本产品的价值须体现在它们尚缺的部分（安全、风控、对账、非托管），而非重复提供收款本身。

**③ 关于"中转站"场景（作为参考，非核心）。** 转售 AI API 的中转站曾被列为候选切入点，但调研显示该场景问题较大：其主体在相当程度上为灰色市场，建立于**凭证盗用、模型替换、数据窃取**之上（白宫警示"数万个欺诈代理账户"，Anthropic 报告单一网络存在 2 万余个欺诈账户），与"愿采用合规通道并付费"的正规客群属**相邻但不同的群体**，且叠加中国合规红线（见第 9 节）。**据此，本产品不应以灰市中转站为核心市场，而应面向上文②所示的正规全球 agent/API 经济。**

**④ 盈利模型薄弱——须重点关注。** 同类产品的费率已被压至极低：**Coinbase x402 每笔仅收 0.001 美元、每月前 1000 笔免费**（当前尚代付 gas）；Stripe 稳定币收单费率 1.5%（但该费率**含法币兑换**，非纯通道费）。据此，**纯支付/结算层几乎无利润空间**，仅靠手续费难以实现正向单位经济，须依赖**增值服务**（对账、路由、合规即服务、法币出入金）盈利，而非纯通道费（详见第 12 节）。

## 9. 合规判断（红黄绿灯）

> 本节对"是否可做、如何做"具有决定性。覆盖**中国、美国、欧盟、香港**四个辖区。
>
> **证据分级与使用边界（重要）**：本节红黄绿灯是**研究备忘录级判断，非产品立项的最终法律依据**。四个辖区的证据强度不一：
> - **中国🔴**：证据最强——有具体文号（银发〔2026〕42 号）、发布日期（2026-02-06）、发文机关（八部委）、关键条文摘录与推理链，可作为"不碰大陆"这一决策的内部依据。
> - **美国🟡**：证据较强——基于 FinCEN 四要素测试（FIN-2019-G001，条文见第 13 节），但"非托管是否豁免"依事实认定、安全港狭窄，**须律师就具体架构确认**，不宜内部先拍"一定豁免"。
> - **欧盟/香港🟡**：证据偏弱——仅确认了"纯自托管豁免 CASP""香港只规管发行人"两个方向性结论，**缺辖区逐条条文比对**，不可外推为"支付中介 整体豁免"，须逐辖区取一线所意见。
>
> **须等律师、不可内部先拍板的结论**：①"我方的具体非托管架构是否豁免美国/欧盟牌照"；②"BVI 及其余辖区的最终选型"；③"中国背景创始人对离岸开户与合作方尽调的实际影响"。这些在第 11、13 节列为待决项。

### 🔴 红灯：中国团队服务大陆客户或触及境内主体，在现行法下明确非法，离岸架构亦无法规避

**有据结论。** 2026 年 2 月 6 日，八部委（央行、发改委、工信部、公安部、市场监管总局、金融监管总局、证监会、外汇局）发布 **银发〔2026〕42 号**《关于进一步防范和处置虚拟货币等相关风险的通知》（取代 2021 年 237 号），明确：

- 虚拟货币相关业务属**非法金融活动**，法币-虚拟货币兑换、币币兑换、作为中央对手方买卖、提供信息中介与定价服务"**一律严格禁止、坚决依法取缔**"；
- "**境外单位和个人不得以任何形式非法向境内主体提供虚拟货币相关服务**"——即境内严禁、境外严管，**无论公司注册地在何处，均不得合法服务大陆客户**；
- 法律后果：行政处罚，构成犯罪的追究刑事责任，**相关民事法律行为无效、损失自担、无法律救济**。

**对本产品的含义**：该红线意味着**无论如何都不能将大陆主体作为客户**，离岸公司与海外服务器均无法规避。**但这并不否定产品命题本身**——本产品面向的是全球 agent 经济，大陆市场只是其中不可触碰的一块。可行做法是**从产品定位上直接排除大陆客群、专注海外**（进入下方黄灯区），而非将中国市场作为核心目标。换言之，这条红线约束的是"服务对象"，不是"能否做这件事"。

### 🟡 黄灯：纯服务海外 + 非托管路线可行，但存在两项硬约束

**约束一：非托管可提供豁免依据，但该依据脆弱、依事实认定，并非"安全港"。**
在美国 FinCEN 框架下，判定主体是否属需持牌的"货币转移商"（代他人转移资金的中介，须注册并履行反洗钱义务），**决定性标准是"是否对资金拥有完全独立控制权"（即是否托管），而非所用技术**。因此，纯非托管、仅被动收款、不代第三方接收并转移资金者，**可能被认定为"用户"而非转移商，从而豁免牌照**。但须注意：

- 该豁免依事实认定、不依主体自称——一旦主体**持有账户余额、代第三方路由资金、或成为支付系统的直接接口**，无论如何自称均将被认定为货币转移商；
- 一项经核验证伪的主张须记录："非托管钱包提供商**绝对**不属于货币转移商"——**已证伪**，即非托管并非免责依据；
- 且**运营 支付中介（结算通道）的主体，其风险显著高于单纯的收款商户**——本产品正属前者。DOJ 在 Tornado Cash 案中已在执法层面突破"非托管即免责"的边界。

**约束二：即便非托管且纯服务海外，亦不免除反洗钱与制裁筛查义务。**
美国 GENIUS Act 项下，FinCEN 与 OFAC 已于 2026.4 发布拟议规则，将稳定币发行方纳入《银行保密法》，须履行反洗钱与 **OFAC 制裁名单筛查**（核对付款地址是否属美国制裁名单）义务。据此，一条务实的稳定币通道即便自认非托管，**仍应主动执行 KYT/地址制裁筛查**；否则一旦资金流向被制裁地址，将面临刑事与声誉风险。

### 🟡 欧盟与香港：对非托管方向有一定支持，但 支付中介 仍可能落网

**欧盟 MiCA（有据结论）**：MiCA（欧盟加密资产统一监管框架）采用"实质重于形式"、以**控制权**判定——**纯自托管**（用户自持私钥、直连去中心化应用、无中心化中介）被明确排除在 CASP（加密资产服务提供商）牌照之外，无需授权与 KYC。这与美国的"控制权"标准一致。**但须注意**：一个实际控制资金或私钥的 支付中介 仍可能落入 CASP 定义、须持牌——"纯自托管豁免"不可外推为"支付中介 业务整体豁免"。

**香港（有据结论，但边界须谨慎）**：香港 2025 年 8 月 1 日实施的稳定币发牌制度，其规管对象**仅为"稳定币发行人"**（发行法币挂钩稳定币须持牌，首两张牌照 2026-04 发给汇丰与 Anchorpoint）；HKMA 官方**未表明**使用稳定币的支付通道/支付中介 需在该框架下另行持牌。**但须注意**：这**不等于**"支付通道在港整体无需牌照"——VASP（虚拟资产服务提供商）或 MSO（金钱服务经营者）牌照仍可能适用。香港能否作为中国团队的可行落脚点，本轮**证据未直接回答，属推测**——须专业意见评估，且仍受前述中国红线约束（不得服务境内主体）。

### 可行路径（部分有据，落地细节仍须律师意见）

综合红黄灯判断，推荐路径为：**彻底剥离中国关联**（无境内主体、不接大陆客户、团队与资金链完全离岸）+ **面向全球 agent 经济、不服务大陆** + **严格执行非托管、反洗钱与制裁筛查**。此路径与本产品命题（面向全球 agent 的自驱支付通道）**完全一致**——大陆本就不是核心市场，排除它并不改变命题，反而使合规路径清晰。

**离岸辖区已有现成参照（有据结论）**：

- **BVI（英属维尔京群岛）**：依 VASP 法案 2022，**发行虚拟资产/稳定币本身不构成受监管活动，仅"托管/控制类"活动才需注册 VASP 牌照**。这为"非托管稳定币通道"提供了现成的免牌照结构范例——与本产品的非托管定位天然契合。
- **Catena Labs（可作反向参照）**：Circle 联创 Sean Neville 创办，融资 3000 万美元 A 轮，正申请美国 OCC 国家信托银行牌照（已受理）。此为**持牌重合规路线**，与"完全离岸免牌照"方向相反，可作对照——说明业内存在"离岸非托管"与"境内持牌"两条并行路径。

**仍须专业意见的部分**："中国背景团队 + 完全离岸 + 纯服务海外"的具体合规架构（离岸控股、创始人身份、资金隔离、KYT/制裁筛查落地）本轮**未查到可复制的公开范例**，须取得专业跨境律师意见，本报告不可作为法律依据。

## 12. 盈利模式分析（分析/推测，非有据结论）

> 前提：第三轮调研**未查到**任何"纯支付层之外靠增值服务实现正向单位经济"的已验证案例，故本节为基于产业逻辑的分析，非有据结论。

### 核心结论：纯通道费不盈利，利润在支付周边的服务；但最赚钱的服务恰是合规风险最大的服务

x402 每笔仅收 0.001 美元、且每月前 1000 笔免费——这是巨头用于获客的引流定价，非利润来源。据此，**盈利不在"转移资金"这一动作本身，而在其周边服务。** 但须正视一处结构性矛盾：**盈利能力最强的业务（法币出入金），恰是合规风险最高、最需牌照的业务**——"能盈利的场景"与"能合规安全落地的场景"存在张力。

### 增值服务盘点（按盈利能力与合规风险排序）

| 增值服务 | 内容 | 商户付费动因 | 盈利能力 | 合规风险 |
|---|---|---|---|---|
| **法币出入金** | 将收到的稳定币兑换为法币提现，或反向充值 | 商户最终以法币结算，属刚需 | 🟢 最高（行业价差普遍 1–3%） | 🔴 最高——正是托管/牌照红线所在 |
| **制裁名单筛查（基础合规）** | 代商户逐笔核对地址是否命中制裁名单 | 法律强制底线，且成本极低（Chainalysis 免费预言机起步） | 🟡 中（可含在基础套餐，本身利润薄） | 🟢 低——本产品为卖方，帮他人合规 |
| **完整 KYT / 风险情报（增强项）** | 风险评分、资金来源分析、行为监测 | 全行业安全层不成熟（本报告核心发现），更深的合规与风控需求 | 🟢 高（按次或订阅，可加价） | 🟢 低——但自身采购第三方 KYT 成本高（年费数十万欧元级） |
| **对账与财务工具** | 链上到账自动匹配订单，出报表、发票、导账 | 加密对账为公认痛点，财务/审计刚需 | 🟡 中（SaaS 订阅，复购稳） | 🟢 低 |
| **agent 风控平台** | 消费限额、熔断、异常拦截、审计 | 本产品技术差异化核心，巨头未做好 | 🟡 中（高级功能/订阅） | 🟢 低 |
| **汇率锁价 / FX 价差** | 法币计价、短时锁价，吃买卖价差 | 商户需价格确定性 | 🟡 中（价差） | 🟡 中 |
| **智能路由 / 垫资** | 跨链跨币选优价、代付 gas、账期垫资 | 降本、改善现金流 | 🟡 中 | 🟡 中（垫资涉信贷） |

### 真正可落地的盈利场景（取"盈利🟢/中 + 合规🟢"交集）

1. **合规与安全中间件（最契合技术优势、风险最低）**：将 KYT 筛查 + agent 风控 + 对账打包为 SaaS，订阅收费。本质是"不做最赚钱的资金转移，而做让他人安全转移资金的中间件"。
2. **法币出入金，但不自持牌照、挂靠持牌方吃分成**：出入金最赚钱亦最危险；不自行承接，而对接持牌合作方，本产品负责获客与体验，收取引荐分成/价差——将最重的牌照风险转移给合作方。
3. **财务运营 SaaS（确定性收入）**：对账、报表、发票、开发者控制台，订阅制。单价不高但复购稳、边际成本低、合规风险极低，构成支撑估值的经常性收入。

### 战略取舍（须由决策层拍板）

存在一处真实两难：**盈利能力最强的业务（自营法币出入金）合规风险最高，而最安全的业务（合规安全中间件）需说服海外客户为尚未放量的需求付费。** 追求短期收入规模会被推向法币出入金（牌照风险大）；追求安全落地则限于"合规 + 风控 + 对账"中间件、面向全球 agent 生态订阅收费（安全，但需承受 agent 支付需求尚小的风险——当前全年规模仅约 7300 万美元）。

**倾向判断**：短期可落地且不触线者，为方案 1、3（合规安全中间件 + 财务 SaaS，订阅制）；方案 2（出入金分成）为将最赚钱部分外包给持牌方的折中。自营出入金虽最赚钱，但牌照负担重，宜在规模验证后再评估。

## 13. 落地参考：离岸非托管合规架构

> 本节将"排除大陆、纯服务海外、非托管"这一方向落成可参照的具体做法。**全节不构成法律意见，正式决策前须咨询跨境专业律师。**

### 核心法律支点：看"控制权"，不看技术标签（有据结论）

美国 FinCEN 2019 年指引（FIN-2019-G001，至 2026 年仍现行）以**技术中立的四要素**判定一个中介是否属于须持牌的"货币转移商"（帮他人搬钱的中介）：①资金归谁所有；②资金存于何处；③所有者是否**直接**与支付系统交互；④**中介是否对资金拥有"完全独立控制权"**。其中 ④ 是决定性要素。结论：**是否需要牌照取决于你是否控制资金，而非你用了什么技术或如何自称。**

### A. 非托管红线清单（可执行的设计约束，非已定论的法律结论）

> **口径说明（与第 9 节一致）**：以下不是"我方已确认豁免牌照"的法律结论，而是**一份设计约束清单**——满足它们可**作为主张"非货币转移商身份"的有利事实**，最终认定须由律师就具体架构确认。

**满足以下全部条件，可作为"非货币转移商"的有利事实（而非豁免的确证）：**

- 绝不持有用户私钥；
- 用户对自己的资金保有完全独立控制、并**直接**与链上交互；
- **不以账户余额形式**为用户记账/持有余额；
- 只提供软件工具（验签、广播交易），不发起、不控制资金流动。

**一旦触及以下任一项，将显著提高被认定为托管/需牌照的风险——通常会被监管视为超出非托管工具的边界（红线，自称"非托管"不改变此风险）：**

| 红线动作 | 说明 |
|---|---|
| 以账户分录形式持有用户余额 | 一旦平台账上有"用户的钱"，即托管 |
| 获得对资金的完全独立控制权 | 含任何"可单方转移资金"的能力 |
| 归集/路由第三方资金再转出 | "接收并再转移"是货币转移的核心特征 |
| 持有私钥（哪怕合同约定只按指令动用） | 托管钱包提供方即基于账户的货币转移商 |
| 把"多签第二密钥"与"托管钱包"服务组合 | 组合后即越出安全港 |

> **重要限定**：该安全港**狭窄**——只覆盖"提供钱包/软件"。一旦做超出此范围的事（归集资金、持钥、控制资金流），即落在安全港之外。

### B. 可照抄的非托管架构范例（有据结论）

- **Coinbase x402（一手规范，可直接参照）**：支付中介 **只对用户签名的支付授权验签并广播上链**，由买方以签名授权标准（USDC/EURC 用 EIP-3009 的 `transferWithAuthorization`、任意 ERC-20 用 Permit2）移动**自己**的资金，支付中介**无法修改金额或收款地址**，仅作交易广播者。协议规范**明文禁止** 支付中介 在客户意图之外移动资金。这正是本产品应采用的设计模式，有公开文档可照抄。
- **Nevermined（单一自述来源，medium）**：以 ERC-4337 智能账户 + 会话密钥实现"结算不托管私钥"，可作架构思路参照，但为自报、无第三方审计。

### C. 反洗钱与制裁筛查：非托管也必须做（有据结论）

本轮有一条主张——"非托管即完全无反洗钱义务"——被**核验证伪**。即：**非托管不等于免除筛查义务**。这里须区分**两个不同层次、成本差一个数量级**的能力，别混为一谈：

- **① 制裁名单筛查（Phase 1 底线，低成本）**：只核对地址是否命中 OFAC 等制裁名单。美国制裁名单以可机读格式发布、支持对数字货币地址编程化查询；**Chainalysis 更提供任何人免费可用、无需建立客户关系的制裁筛查预言机（智能合约）**——是几乎零成本的基础入口，可放在下单/结算关键路径。**这是一期就该做的合规底线。**
- **② 完整 KYT（后续增强，高成本，范围更宽）**：风险评分、资金来源追溯、行为监测等，远超"是否命中名单"。须用付费产品（Chainalysis 付费 KYT、TRM、Elliptic，量级参考第一轮：Chainalysis 约 €120k–250k/年、Elliptic 约 €80k–180k/年，TRM 未公开）。**这不是能像制裁筛查那样低成本进一期的东西，须按需单独评估。**

> **重要提醒**：文档其余处提到"筛查"时，一期指的是 ①（制裁名单筛查），而非 ②（完整 KYT）——不要因"筛查已接入"就以为完整 KYT 也已低成本覆盖。

### D. 离岸辖区（仅 BVI 有据，其余须补）

- **BVI（英属维尔京群岛，有据）**：VASP 法案**只监管"代表第三方"提供的托管/兑换/转移服务**；**非托管钱包**及"仅接受加密货币作为自身商品/服务付款"被列为**免于 VASP 注册的豁免类别**。软件型、非托管、用户独占控钥的业务落在受监管定义之外——与本产品定位天然契合。（经 Walkers、Mourant 等一线离岸所佐证；"支付 支付中介 在 BVI 必须注册 VASP"的反向主张已被 0:3 证伪。）
- **行业通行实践（medium）**：支付/稳定币提供商通过"**本地不托管用户资金**"避免成为本地 VASP；纯基础设施型提供商更进一步，**把反洗钱义务下移给终端客户**（让终端客户成为负责 AML 的 VASP）。
- **⚠️ 其余辖区未查实**：开曼、新加坡、瑞士、阿联酋（迪拜 VARA）、香港、爱沙尼亚/立陶宛对"非托管支付 支付中介"的具体要求、设立成本与周期、声誉风险，以及欧盟 MiCA"控制权"标准的细节，本轮**未充分验证**，不宜据本文下结论，须逐辖区取一线所意见。

### E. 最大风险（须正视）

- **安全港狭窄且依事实认定**：产品一旦为改善体验而"顺手"做了归集、垫资、持钥，即可能滑出非托管、触发牌照。**产品设计须把"非托管红线"作为硬约束持续守住。**
- **执法边缘风险**：Tornado Cash 案对"非托管 + 匿名化"制造了执法张力；本产品的普通支付载荷与之不同，但边界存在被执法突破的风险。
- **政策变动**：FinCEN 指引为解释性文件（非成文法），可被行政政策或诉讼改变。
- **未查到的两项经营变量**：银行/法币出入金合作方是否接单、以及**创始团队中国背景对开户与合作方尽调的实际影响**，本轮**未查到证据**，须实地验证——这可能是比法律条文更现实的落地障碍。

### 小结

法律路径是**清楚且可执行**的：以"控制权"为纲、守住非托管红线、照抄 x402 的验签+广播设计、接入 OFAC + Chainalysis 免费预言机做筛查、辖区以 BVI 为起点。真正尚未落定的是**辖区细化对比**与**银行/出入金合作方及中国背景创始人的实际可行性**——这两项须律师意见 + 实地接触验证。

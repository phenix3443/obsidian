# Agent Rules

## 通用规则

### 任务规划

- 修改代码前，必须先制定方案；但简单代码改动可跳过方案。
- 在方案落盘前，不得开始实际代码修改。
- 方案必须先保存到当前 workspace 的 `tasks/` 目录下，优先放在对应任务子目录或独立任务文件中。
- 仅当任务本身是补充或调整 `AGENTS.md` 工作流程时，不要求额外创建方案文件。
- 简单代码改动可跳过创建方案、issue 和 PR，直接在当前分支提交；其中实际代码改动少于 5 行时，可直接在 `main` 分支用一个 commit 完成改动并推送远端。
- 实际代码改动不少于 5 行时，必须先创建对应的 issue 和 PR，再在对应 PR 中实施改动。
- 对 `clash` 和 `clash-rules` 两个仓库，允许直接在 `main` 分支修改、提交并推送，不要求创建 issue 或 PR。
- 当用户要求“记住某件事”时，必须先判断这是否属于长期有效的 workspace / 仓库协作规则。
- 若属于长期流程、提交规范、文档约束、评审约束或其他稳定规则，应同步写入对应的 `AGENTS.md`；若只是当前任务的临时上下文或一次性偏好，则不写入。

### Git 工作流

- 当创建 PR（包括 draft PR）后，必须立即进入 CR（代码审查）循环，不能把“PR 已创建”视为任务完成。
- `clash` 和 `clash-rules` 两个仓库在直接修改 `main` 分支时，不要求创建 PR，因此不适用基于 PR 的 CR 循环、checks 触发和 PR 收尾规则；此时只需完成本地自审与必要验证后直接提交推送。
- CR 循环至少包括：
  1. 检查 PR 的 review、review comment、普通评论和 checks 状态。
  2. 处理所有明确可执行的问题后再次 push。
  3. 每次 push 后重新检查 review、评论和 checks。
  4. 直到没有新的可执行 CR 问题，且 checks 全部通过或明确确认无 checks 属于预期为止。
- 如果 PR 创建后没有任何 checks 被触发，默认视为流程缺陷，必须优先修复触发条件、工作流配置或其他阻塞，而不是直接宣布“已完成 CR”。
- 如果暂时没有外部 reviewer 或评论，也必须至少做一次针对 diff 和 CI 触发状态的自审，确认不存在待处理问题后，CR 才算完成。
- 只要 PR 上仍有 failed / pending checks、未处理的 review 意见，或新增评论尚未响应，就不能结束该任务。
- PR 合并完成后的标准收尾动作为：将对应仓库本地分支切回 `main`，同步最新远端代码，并清理已合并的本地功能分支与远程功能分支。
- 当用户说“收尾这个 issue”时，默认必须依次执行：
  1. 更新对应 issue 和 PR 的标题、正文、必要评论，使其反映当前分支最新代码变化。
  2. 自动合并对应 PR，并关闭对应 issue。
  3. 执行上述 PR 合并后的标准收尾动作。
- issue 和 PR 的标题、正文、评论中禁止出现本地绝对路径；如需引用方案或文档，只能使用仓库相对路径、GitHub 文件链接，或直接概述内容。

## Workspace 相关规则

### Workspace 目录结构

- 当前目录 `clash/`：存放 workspace 级文档、计划和 `clash.code-workspace` 配置。
- `clash`：私有主仓库，处理真实配置、私有脚本、实例运行逻辑和敏感数据。
- `mihomo-companion`：公开仓库，处理通用 CLI、模板、示例配置、公开文档和发布流程。
- `clash-rules`：规则仓库，处理域名分流、代理命中、直连/代理策略。

### 命令执行上下文

- 当前 workspace 根目录 `clash/` 是 workspace 容器目录，只存放 workspace 级文档、计划和 `clash.code-workspace` 配置，不是任何业务仓库的代码根目录，也不承载 `mihctl`、`Makefile`、`go.mod`、`package.json` 等仓库级执行入口。
- 不得仅因当前目录名为 `clash`，就假设自己已经处于私有主仓库；必须先结合 `clash.code-workspace` 和当前 workspace 目录结构判断命令所属仓库。
- 在运行 `./mihctl`、`make`、`go test`、脚本或其他仓库级命令前，必须先切换到对应挂载仓库，不能先在 workspace 根目录试探性执行再根据报错回退。
- 如果用户给出的命令缺少仓库前缀，默认先按代码归属判断目标仓库：私有实例运行命令进 `clash`，通用 CLI / 模板 / 发布相关命令进 `mihomo-companion`，规则更新相关命令优先进 `clash-rules`。

### 测试执行约束

- 进行测试时，默认优先使用 Docker 或其他隔离容器环境执行，避免直接影响宿主机器的网络环境。
- 尤其是会修改路由、TUN、iptables、防火墙、网络命名空间、代理端口或系统服务状态的测试，禁止直接在宿主机网络环境中执行；若确需在宿主机验证，必须先得到用户明确确认。

### 多仓 Git 工作流

- 当用户明确说 `commit & push` 时，必须遍历当前 workspace 下除 `clash-task` 之外的所有仓库。
- 对每个仓库分别检查未提交改动；有改动就分别提交并推送，不能只处理当前仓库。
- 如果某个仓库存在冲突、明显不应提交的脏文件，或其他会影响安全提交的问题，先向用户指出具体仓库和阻塞原因，再决定是否继续。

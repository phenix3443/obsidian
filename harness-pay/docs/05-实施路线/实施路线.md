# 实施路线

## Phase 1：跑通基座

- 部署 Polar 测试环境。
- Dashboard、Checkout、Portal 中文化。
- 清理 Polar 品牌和默认 onboarding。
- 完成商品创建与测试卡支付演示。

## Phase 2：Web2 支付闭环

```text
商户入驻审核 -> 建商品 -> 欧美买家卡支付
-> 权益发放 -> 账本可见净额 -> 商户 payout
```

依赖法律主体、Stripe 账户、美欧银行账户、Stripe Tax 和商户 KYB。

## Phase 3：Web3 支付闭环

```text
USDC 支付 -> 链上确认 -> 权益发放 -> KYT
-> 归集兑换 -> 多币种账本 -> payout
```

验收重点：完成黑名单地址拦截，连续 7 天无未解释四方对账差异，少付/超时等异常不造成资损。

## Phase 4：Agent 原生接入

按协议级需求和客户需求触发，逐步评估 X402/AP2、直连钱包、扩链扩币和移动端。

## 当前阻塞点

1. 法律主体、Stripe 收单账户和美欧银行账户。
2. 托管签名方案。
3. 兑换服务商选型。
4. Crypto 分账方案的法务复核。
5. zkMe 的 KYB/KYT 能力边界。
6. 大陆商户法币入境与稳定币结算路径。
7. 非美欧买家和税务范围。
